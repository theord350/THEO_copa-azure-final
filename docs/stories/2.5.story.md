## Story

**Como** aluno do workshop "Living Lab Azure-Native",
**eu quero** completar a Fase 5 em 8h expondo 3 tools MCP (`consultar_disponibilidade`, `verificar_ingresso`, `consultar_bracket`) via Function .NET e integrando um componente React de chatbot no frontend conversando com Gemini 2.0 Flash,
**para que** eu compreenda o protocolo MCP, integração LLM em produtos reais, e tenha um chatbot que responde perguntas reais consultando o SQL do FIFA 2026 Tickets — com portabilidade entre LLMs (Gemini → Groq → Mistral via env var).

## Acceptance Criteria

1. **AC-1 — Branch criada:** `phase-05-ai-mcp` partindo de `phase-04-orchestration`, com workflow `.github/workflows/deploy-phase-05.yml` seguindo o pattern dos anteriores.

2. **AC-2 — MCP Server implementado em Function .NET dedicada** — `src/Fifa2026.V2.McpServer/` com endpoint `/mcp` JSON-RPC 2.0 (conforme spec modelcontextprotocol.io). Separada das Functions de compra (`src/Fifa2026.V2.Functions/`).

3. **AC-3 — Tool `consultar_disponibilidade`:** recebe `matchId` (integer) e/ou `matchDescription` (string, ex.: "Brasil x Argentina"), retorna `{ vipDisponivel, cat1Disponivel, cat2Disponivel, precoVip, precoCat1, precoCat2 }` consultando o SQL (tabela `purchases`, `ticket_categories` ou equivalente conforme schema real do projeto).

4. **AC-4 — Tool `verificar_ingresso`:** recebe `ingressoId` (string ou integer — o QR code ou ID do ingresso), retorna `{ valido, comprador, partida, categoria, dataCompra }` consultando o SQL.

5. **AC-5 — Tool `consultar_bracket`:** recebe `rodada` (string, ex.: "oitavas", "quartas"), retorna jogos + placares + classificados conforme tabela de matches no SQL.

6. **AC-6 — Pinning do MCP SDK documentado:** @architect produz `docs/architecture/ade-002-mcp-pinning.md` com versão específica do SDK (nuget/npm), justificativa e plano de upgrade. Esta ADE é **pré-condição** para o @dev iniciar a implementação do MCP Server (Task 2).

7. **AC-7 — Componente React de chatbot** em `Lovable/World Cup Tickets Hub/src/components/Chatbot.tsx` com UI mínima (input text + histórico de mensagens + botão enviar). Usa os padrões shadcn/ui existentes no projeto (confirmar componentes disponíveis em `Lovable/World Cup Tickets Hub/src/`).

8. **AC-8 — Integração Gemini 2.0 Flash:** chatbot envia mensagens para Gemini com a lista de tools MCP disponíveis; Gemini chama MCP server quando precisa de dados; resposta do MCP retorna ao Gemini que gera resposta natural ao usuário. Chamadas ao MCP Server roteadas **via gateway YARP** (Bearer token Entra do usuário logado — herança de F3).

9. **AC-9 — Identificação do usuário via entraOid:** o gateway YARP propaga `X-Entra-OID` para o MCP Server (padrão estabelecido em F3, confirmado em `src/Fifa2026.V2.Gateway/Program.cs` transform). O MCP Server pode usar o `entraOid` para personalizar respostas ou logar a identidade do usuário chamante — sem revalidar o JWT (o gateway é o guardião).

10. **AC-10 — Portabilidade via env var:** trocar `LLM_PROVIDER=gemini|groq|mistral` no ambiente do frontend/chatbot faz o chatbot funcionar com qualquer um dos três providers sem mudança de código. Demonstrar com Groq como plano B documentado.

11. **AC-11 — Smoke tests didáticos** (3 perguntas canônicas validadas ao vivo):

    - "Esse ingresso ID 123 é válido?" → Gemini chama `verificar_ingresso`
    - "Quem está nas oitavas?" → Gemini chama `consultar_bracket`

12. **AC-12 — Fallback documentado para Gemini cair:** SPEAKER-NOTES incluem cache de respostas pré-carregado + procedimento de troca emergencial para Groq (troca a env var `LLM_PROVIDER`).

13. **AC-13 — 6 artefatos didáticos** em `docs/workshops/phase-05/`:

    - `PORTAL-GUIDE.md` (Function MCP Server provisioning, Gemini API key via App Setting)
    - `SPEAKER-NOTES.md` + seção bônus "portabilidade entre LLMs"
    - `slides.md`

    - Branch + workflow CI/CD

14. **AC-14 — Roteiro de 8h respeitado:** blocos conforme blueprint F5 (Conceitos MCP/LLM, MCP Server Function, chatbot React, integração Gemini, smoke tests, bônus portabilidade, retro). Documentado em SPEAKER-NOTES.

## Dev Notes

### Arquitetura técnica delta

```
[Browser — frontend Vite]
   │ usuário digita "Tem ingresso pra Brasil x Argentina?"
   ▼
[Chatbot.tsx (React)]
   │ useLlmChat('gemini')
   ▼
[Gemini 2.0 Flash API]
   │ recebe pergunta + tools list (MCP schema)
   │ decide chamar consultar_disponibilidade
   ▼
[Gateway YARP — Container App por aluno]          ← OBRIGATÓRIO (herança de F2/F3)
   │ valida Bearer token (AddJwtBearer)
   │ propaga X-Entra-OID
   ▼
[MCP Server Function: POST /mcp JSON-RPC]
   │ tools/call → consultar_disponibilidade(matchId)
   │ lê X-Entra-OID para logging (não revalida JWT)
   │ Dapper SELECT em SQL (purchases, matches)
   ▼
[Gemini]
   │ recebe resultado da tool
   │ gera resposta natural
   ▼
[Chatbot UI]
   │ exibe resposta ao usuário
```

**Nota importante:** o MCP Server é uma Function .NET acessível VIA o gateway YARP — não direto do browser. Isso mantém o gateway como ponto único de autenticação/autorização (ADE-004/ADE-005 Inv 4) e é o caminho pedagógico correto ("tudo passa pelo gateway"). O chatbot envia o Bearer token do usuário logado (obtido via MSAL.js em F3).

### MCP tools schema (JSON Schema — rastreado a modelcontextprotocol.io)

```json
{
  "name": "consultar_disponibilidade",
  "description": "Consulta disponibilidade e preços de ingressos para uma partida da Copa 2026",
  "inputSchema": {
    "type": "object",
    "properties": {
      "matchId": { "type": "integer", "description": "ID numérico da partida" },
      "matchDescription": { "type": "string", "description": "ex: Brasil x Argentina" }
    },
    "required": []
  }
}
```

Pattern análogo para `verificar_ingresso` (recebe `ingressoId`) e `consultar_bracket` (recebe `rodada`).

### Padrão de acesso a dados

O MCP Server segue o padrão de acesso a dados já estabelecido em `src/Fifa2026.V2.Functions/Data/` (Dapper, `IPurchaseRepository`). O `@dev` deve criar interfaces análogas em `src/Fifa2026.V2.McpServer/Data/` para as queries de disponibilidade, ingresso e bracket — mantendo o padrão do projeto.

### Conteúdo bônus: portabilidade entre LLMs

SPEAKER-NOTES inclui demo de 5min: aluno troca `LLM_PROVIDER=gemini` → `groq` → `mistral` e a mesma pergunta produz respostas equivalentes via o mesmo MCP Server. Demonstra que o protocolo MCP desacopla o LLM dos dados.

### Troubleshooting conhecido

| Sintoma | Causa provável | Solução |
|---|---|---|
| Gemini não encontra as tools | `tools/list` retorna formato inválido | Validar schema JSON-RPC contra spec MCP; conferir Content-Type: application/json |
| 401 ao chamar MCP Server | Bearer token não enviado ou expirado | Verificar MSAL.js renova token antes de expirar; relogar se necessário |
| X-Entra-OID ausente no MCP Server | Gateway não propagou (token não autenticado) | Confirmar usuário está logado; gateway propaga OID apenas para requests autenticados |
| Timeout de LLM (> 10s) | Groq/Gemini latência ou quota | Usar fallback conforme AC-12; mostrar loading state no UI |
| Tool call não dispara | LLM não reconheceu intenção de consulta | Ajustar description da tool (mais contexto sobre quando usar) |

### Testing

- Smoke: ponta-a-ponta com timeout 10s por pergunta
